Java 例外処理〜実行時のトラブルを読み解き、扱う〜

1. 本単元の概要

プログラムの実行中に起きる予期せぬトラブルを Java では例外（Exception）として表し、try-catch などで扱うことで、いきなり強制終了せずにログや代替処理につなげられます。

このテキストは、SQL / JDBC で扱った items（商品）のイメージを軸にしつつ、try-catch-finally・複数 catch・try-with-resources までを図と短いコードでたどります。発展として、社員と部署（別テーブル想定）のチェック例外も扱います。

※ 図は diagrams/Exception/ の PNG を相対パスで参照します。ファイルがまだ無い環境では画像が表示されないことがあります。

2. 学習目標

次の項目を、図やコードとあわせて説明できることを目指します。

コンパイルエラーと実行時のトラブルを区別できる

コンパイルは「文法と型のチェック」、実行中の例外は「動いているときに起きた異常」として切り分けられる。

チェック例外と非チェック例外をざっくり説明できる

チェック例外は呼び出し側に処理を強いることが多い。非チェック例外（RuntimeException 系）は実行時に起きやすい論理ミス・前提崩れとして扱われる。

try / catch / finally の流れを追える

try で囲んだ範囲で例外が起きると、対応する catch にジャンプする。finally は（ほぼ）必ず通る。

複数 catch で広い型（Exception）を最後に置く理由を言える

上から順に「この型か？」と照合するため、親クラスを先に書くと子の catch に届かない。

multi-catch（|）の役割を言える

同じ処理で複数種類の例外をまとめて捕まえたいときに使う（catch パラメータは事実上 final 扱い）。

try-with-resources が何のためか一言で言える

開いたリソース（close が必要なもの）を、ブロックの終わりで確実に閉じるための書き方。

3. 本編

3-1. 例外（Exception）とは

例外は、実行中に発生したトラブルを表すオブジェクトです。捕捉しないとスタックトレースを出してスレッドが終了する、などの動きになります（どう終わるかは状況次第）。

発生タイミング 実行前（ビルド時） プログラムの実行中
主な原因 文法ミス、型の不一致、未定義の名前など 不正な入力、0 除算、ファイル欠損、前提が崩れた参照など
対処の方向け ソースコードの修正 try-catch、throws、入力検証、設計の見直しなど

3-4. チェック例外（Checked）

厳密には「Exception のサブクラスであって RuntimeException 系ではないもの」がチェック例外です。DB やファイルなど外の世界と話すと現れやすく、try-catch か throws で処理の経路を書かないとコンパイルが通らない、という性質を持ちます。

例：IOException、SQLException（items を JDBC で扱うときに登場します）。

図4：チェック例外のイメージ

3-5. 非チェック例外（Unchecked）

RuntimeException とそのサブクラスです。多くは前提の崩れ・バグに近いので、呼び出し側に try-catch を強制しません（書いてもよい）。

例：NullPointerException、IllegalArgumentException、ArithmeticException。

図5：非チェック例外のイメージ

3-6. エラー（Error）

Error はメモリ枯渇（OutOfMemoryError）など、アプリ単体では立て直しが難しいことが多い層です。入門では捕まえない方針でよいです。

図6：Error の位置づけ

3-7. なぜそう分かれるか（責任の分かれ目）

「外因で起きやすい／処理経路を書かせる」vs「バグ寄り／強制しない」という設計上の役割で整理すると、頭に入りやすくなります。

図9：正常系と異常系の実行フロー

構文の流れだけ、最短の例（0 除算）で確認します。items の話はこのあとの節で続けます。

finally は例外の有無にかかわらずほぼ必ず実行されます（System.exit で JVM が止まるなど、例外は別の機会で触れれば十分です）。

public class DivideSample {

	public static void main(String[] args) {
		try {
			int result = 10 / 0;
			System.out.println("結果=" + result);
		} catch (ArithmeticException e) {
			System.out.println("エラー：0 で割ることはできません。");
		} finally {
			System.out.println("処理を終了します。（finally）");
		}
	}
}

3-9. EC の items 例：カートに入れるときの検証（非チェック）

SQL の items テーブル（item_id、item_name、unit_price、在庫イメージの数）に合わせた小さな Java 側のモデルです。個数が不正、または在庫不足なら IllegalArgumentException を投げます（非チェック）。

public class ShopItem {

	// フィールド（items テーブルに対応するイメージ）
	int itemId;
	String itemName;
	int unitPrice;
	int stock;

	public ShopItem(int itemId, String itemName, int unitPrice, int stock) {
		this.itemId = itemId;
		this.itemName = itemName;
		this.unitPrice = unitPrice;
		this.stock = stock;
	}

	// カートに載せる＝在庫を減らす前提の簡易処理
	public void addToCart(int quantity) {
		if (quantity <= 0) {
			throw new IllegalArgumentException("個数は1以上を指定してください");
		}
		if (stock < quantity) {
			throw new IllegalArgumentException("在庫が足りません");
		}
		stock = stock - quantity;
		System.out.println(itemName + "（item_id=" + itemId + "）を" + quantity
				+ "点。単価" + unitPrice + "円");
	}
}

3-10. 在庫出荷とチェック例外（OutOfStockException）

倉庫側だけを切り出した ItemStock です。在庫不足のときはチェック例外にし、呼び出し側に try-catch か throws を書かせます（業務ルールとして「必ず処理させたい」場面の例）。

public class OutOfStockException extends Exception {

	public OutOfStockException(String message) {
		super(message);
	}
}

public class ItemStock {

	int stock;

	public ItemStock(int initialStock) {
		this.stock = initialStock;
	}

	public void ship(int quantity) throws OutOfStockException {
		if (quantity <= 0) {
			throw new IllegalArgumentException("出庫数は1以上を指定してください");
		}
		if (stock < quantity) {
			String message = "在庫不足（必要:" + quantity + " 実在庫:" + stock + "）";
			OutOfStockException ex = new OutOfStockException(message);
			throw ex;
		}
		stock = stock - quantity;
		System.out.println("出荷OK。残り在庫=" + stock);
	}
}

3-11. try / catch / finally（CatalogDemo）

商品行と倉庫を続けて試す main です。

public class CatalogDemo {

	public static void main(String[] args) {
		ShopItem lineItem = new ShopItem(1, "水", 100, 3);
		try {
			lineItem.addToCart(10);
		} catch (IllegalArgumentException e) {
			System.out.println("カートに入れられません: " + e.getMessage());
		} finally {
			System.out.println("カート処理ブロックを抜けます（finally）");
		}

		ItemStock warehouse = new ItemStock(2);
		try {
			warehouse.ship(1);
			warehouse.ship(5);
		} catch (OutOfStockException e) {
			System.out.println("出荷不可: " + e.getMessage());
		}
	}
}

3-12. 発展：社員と部署 ID（関連テーブル想定・チェック例外）

employees が departments の主キーに外部キーで紐づく、というイメージです。ここでは簡略化し、「有効な部署 ID は 1〜5 だけ」と決め打ちして検証します。

public class UnknownDepartmentException extends Exception {

	public UnknownDepartmentException(String message) {
		super(message);
	}
}

public class EmployeeOnboarding {

	public void assignDepartment(int departmentId) throws UnknownDepartmentException {
		if (departmentId < 1 || departmentId > 5) {
			String message = "存在しない部署ID: " + departmentId;
			UnknownDepartmentException ex = new UnknownDepartmentException(message);
			throw ex;
		}
		System.out.println("部署ID " + departmentId + " に配属しました（departments テーブル想定）");
	}
}

3-13. 複数 catch の順序と multi-catch（|）

順序：より具体的な型を上に、Exception のような広い型は最後。先に親を書くと子の catch は到達不能になりコンパイルエラーになります。

multi-catch：同じ復旧処理でよいとき、型を | で並べます。

public class MultiCatchSample {

	public static void main(String[] args) {
		String text = null;
		try {
			System.out.println(text.length());
		} catch (NullPointerException | IllegalArgumentException e) {
			System.out.println("引数や参照が不正: " + e.getClass().getSimpleName());
		} catch (Exception e) {
			System.out.println("その他: " + e.getMessage());
		}
	}
}

3-14. コラム：変数のスコープと finally

try の中だけで宣言した変数は、そのブロック外（finally 含む）では使えません。try の前で宣言し、try 内で代入するのが定石です。

// NG のイメージ：finally から try 内の totalPrice は参照できない
// try { int totalPrice = unitPrice * qty; } finally { ... totalPrice ... }

OK 例

int totalPrice = 0;
try {
	totalPrice = 100 * 2;
} finally {
	System.out.println("合計（参考）=" + totalPrice);
}

3-15. try-with-resources（close を自動で呼ぶ）

AutoCloseable を実装したオブジェクトを try の丸括弧内で宣言すると、ブロックを抜けるときに close() が呼ばれます。JDBC の Connection なども同じ考え方で閉じます（ここではコードを短くするためファイル出力で示します）。

public class LogResource implements AutoCloseable {

	public void writeLine(String line) {
		System.out.println("[LOG] " + line);
	}

	@Override
	public void close() {
		System.out.println("[LOG] リソースを閉じました");
	}
}

public class TryWithResourcesSample {

	public static void main(String[] args) {
		try (LogResource log = new LogResource()) {
			log.writeLine("処理中");
		}
	}
}

5. まとめ

  • 目的：予期せぬエラーでも、ログや代替処理につなげ、必要なら安全に終了する。
  • catch の順序：具体的なクラス（子）から、抽象的なクラス（親）へ。
  • スコープ：finally でも読みたい変数は try の前に宣言する。
  • 自動で閉じる：close() が必要なリソースは try-with-resources を使う（DB 接続も同じ考え方）。
try { 危ない処理 }
catch (具体的な型 e) { 復旧やログ }
catch (Exception e) { 最後の網 }
finally { 後始末（ほぼ必ず） }

try (リソース r = ...) { 処理 }  // 終わりで r.close()
