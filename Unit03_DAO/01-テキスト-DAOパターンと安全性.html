第3回：DAOパターンと安全性〜「直書きJDBC」からの脱却〜

1. 今日は「DAO」を固定する（概要）

この回は、Java から DB に命令する作業を **DAO（DB担当窓口）** に集約し、上流側（呼び出し側）と切り離すための回です。

扱う題材の基本は items（ECサイトの商品）で、必要になったときだけ発展として関連テーブルを扱います。

2. なぜDAOが必要？（直書きJDBCの弱点）

要点

  • SQLやDB処理を呼び出し側に散らすと、修正のたびに影響範囲が増えやすい（変更に弱い）
  • 文字列連結でSQLを組み立てると、意図しない入力が混ざりやすい（攻撃に弱い）

3. DAO / DTO / Entity の3箱（写経）

写経（Service → DAO → DB） 
Service → DAO → DB
DTO = 運ぶ箱（画面/呼び出し向け）
Entity = DBの行（テーブル向け）

4. データの旅（保存・取得）

ポイントは、呼び出し側は「DAOに頼むだけ」、DAOが PreparedStatement と ResultSet を責任を持って扱うことです。

5. PreparedStatement：? と setXxx の対応

最小コードイメージ（items検索） 
String sql = "SELECT item_name, unit_price FROM items WHERE category = ?";

try (Connection con = DriverManager.getConnection(URL, USER, PASS);
     PreparedStatement ps = con.prepareStatement(sql)) {
	ps.setString(1, category); // 先頭の「?」に値を入れる

	try (ResultSet rs = ps.executeQuery()) {
		while (rs.next()) {
			// rs.getString(...), rs.getInt(...)
		}
	}
}

6. 最小のミッション（DAOへ切り出す）

  • 直書き JDBC を DAO クラスに移し、呼び出し側は DAO を「1行呼ぶ」状態にする
  • SQL は PreparedStatement にし、文字列連結をやめて「?」へ setXxx で値を渡す
