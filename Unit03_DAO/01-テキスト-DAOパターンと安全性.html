第3回：DAOパターンと安全性〜「直書きJDBC」からの脱却〜

1. 今日は「DAO」を固定する（概要）

この回は、Java から DB に命令する作業を **DAO（DB担当窓口）** に集約し、上流側（呼び出し側）と切り離すための回です。

扱う題材の基本は items（ECサイトの商品）で、必要になったときだけ発展として関連テーブルを扱います。

2. なぜDAOが必要？（直書きJDBCの弱点）

要点

  • SQLやDB処理を呼び出し側に散らすと、修正のたびに影響範囲が増えやすい（変更に弱い）
  • 文字列連結でSQLを組み立てると、意図しない入力が混ざりやすい（攻撃に弱い）

プログラミング学習における写経は、ソースコードを見たまま打つことで、頭でなく体で覚える学習方法という理解でお願いします。下のサンプルは Eclipse などにそのまま写して動かす前提です。

5. DAOクラスの基本型（DaoBase・ItemDto・ItemDao）

（参考）Entity と DTO をクラスとして分ける場合

  • メリット：例として、パスワードのハッシュなど、画面や公開 API に載せたくない列は、DB に近い側（いわゆる Entity 側）にだけ保持し、外に運ぶ DTO には含めない、といった切り分けがしやすい。
  • デメリット：クラスやマッピング（詰め替え）が増え、名前の使い分けも悩みやすくなる。

大規模なアプリではこのように分けることがありますが、本講義規模の JDBC では次の ItemDto（JavaBeans）だけで十分です。

  • 接続情報は設計にもよるが、ここでは親クラス DaoBase に getConnection() を用意して集約する。
  • テーブルごとに1クラス：例として items 用に ItemDao を用意する。
  • CRUD：1件検索・全件検索・1件挿入・1件更新・1件削除を ItemDao に用意する（参照は ItemDto を返す、更新は件数を返す）。
  • 残りのメソッド（findAll / update / delete など）の穴埋めは、練習問題で完成させる。

6. データの旅（保存・取得）

ポイントは、呼び出し側は「DAOに頼むだけ」、DAOが PreparedStatement と ResultSet を責任を持って扱うことです。

7. PreparedStatement：? と setXxx の対応

最小コードイメージ（items検索） 
String sql = "SELECT item_name, unit_price FROM items WHERE category = ?";

try (Connection con = DriverManager.getConnection(URL, USER, PASS);
     PreparedStatement ps = con.prepareStatement(sql)) {
	ps.setString(1, category); // 先頭の「?」に値を入れる

	try (ResultSet rs = ps.executeQuery()) {
		while (rs.next()) {
			// rs.getString(...), rs.getInt(...)
		}
	}
}

8. 最小のミッション（DAOへ切り出す）

  • 直書き JDBC を DAO クラスに移し、呼び出し側は DAO を「1行呼ぶ」状態にする
  • SQL は PreparedStatement にし、文字列連結をやめて「?」へ setXxx で値を渡す
