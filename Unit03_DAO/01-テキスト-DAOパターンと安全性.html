1. 今日は「DAO」を固定する（概要）

この回は、Java から DB に命令する作業を DAO（DB担当窓口） に集約し、上流側（呼び出し側）と切り離すための回です。第3回は参照系（SELECT）のみ扱います。

題材は emps（社員）と depts（部署）です（環境構築 HTML の lesson DB）。単表の検索から、2テーブル結合（JOIN）まで段階的に進めます。

INSERT / UPDATE / DELETE とトランザクションは 第4回（Unit04_DAO_Update）で扱います。

2. なぜDAOが必要？（直書きJDBCの弱点）

要点

  • SQLやDB処理を呼び出し側に散らすと、修正のたびに影響範囲が増えやすい（変更に弱い）
  • 文字列連結でSQLを組み立てると、意図しない入力が混ざりやすい（攻撃に弱い）

【流れの把握】並べ替え（紛らわしい文つき）

問題1　次の ア〜コ のうち、ItemDao#findById（参照系・1件取得）として実際に行う処理だけを取り出し、実行される順番に並べてください（例：ウ→ア→… のように記号の列をノートに書く）。使わない文が4つ残ります。さらに、残った4つを「更新系APIの誤用」「順序が不可能」「トランザクションの混同」のどれかに分類し、理由を一言で書いてください。

  • ア　SQL 文字列を用意する（SELECT item_id, category, item_name, unit_price FROM items WHERE item_id = ?）
  • イ　getConnection() でデータベースへの接続を取得する
  • ウ　prepareStatement(sql) で SQL をドライバに渡し、実行準備（プリコンパイル）をする
  • エ　setInt(1, itemId) でプレースホルダに検索キーを格納する
  • オ　executeQuery() で検索を実行し、得られた ResultSet を開く
  • カ　rs.next() でカーソルを進め、行があれば getXxx で列を読み ItemDto を組み立てて返す（なければ null）
  • キ　検索を executeUpdate() で実行し、その戻り値を ResultSet に格納する
  • ク　INSERT 文を executeUpdate し、続けて同じ ResultSet で SELECT 結果を読む
  • ケ　接続を取得する前に、ResultSet のカーソルで item_name を読み取る
  • コ　commit() でトランザクションを確定してから、SQL 文字列を用意する

public ItemDto findById(int itemId) {
	String sql = "SELECT item_id, category, item_name, unit_price FROM items WHERE item_id = ?";

	// （ここ①：この try がまとめて行っていること）
	try (Connection con = getConnection();
			PreparedStatement ps = con.prepareStatement(sql)) {

		// （ここ②：SQL の「?」と対応づける操作）
		ps.setInt(1, itemId);

		// （ここ③：内側の try が担うこと）
		try (ResultSet rs = ps.executeQuery()) {
			// （ここ④：ResultSet のカーソルと1行分のデータ）
			if (rs.next()) {
				return new ItemDto(
						rs.getInt("item_id"),
						rs.getString("category"),
						rs.getString("item_name"),
						rs.getInt("unit_price"));
			}
			// （ここ⑤：該当行が無いとき）
			return null;
		}
	}
}

ヒント①

並べ替えでは「参照は executeQuery」「更新件数は executeUpdate」を切り分けます。接続より先に ResultSet はあり得ません。

記述①は「誰が誰を継承し、何を返すメソッドか」。記述③の①〜⑤は、接続→文の準備→プレースホルダ→実行→カーソル、の五段に対応します。

ヒント②

外側の try-with-resources は getConnection() と prepareStatement(sql) をまとめて閉じます。

setInt(1, itemId) の 1 は SQL 内のプレースホルダの先頭です。

解答例

【解説・確認用】

問題1　正しい実行順は ア → イ → ウ → エ → オ → カ。キは参照に executeUpdate は使わず、戻り値も ResultSet ではありません。クは INSERT と SELECT が混ざった誤ったイメージです。ケは接続前に ResultSet を読めません。コは単純な1件取得では不要な順序で紛らわしいです。

問題2　正解は C（ItemDto）。ItemDao は窓口、DaoBase は接続の共通部です。

記述②の模範コメント（ここ①〜⑤）

  • ①　接続を取得し、実行する SQL 文をバインドする（try-with-resources）
  • ②　先頭のプレースホルダに itemId をセットする
  • ③　検索を実行し、結果セット（カーソル）を開く
  • ④　カーソルを1行進め、行があれば列から値を読み取り ItemDto を組み立てて返す
  • ⑤　該当行がなければ null を返す

記述①・②・③（コメント付きメソッド全文）

public ItemDto findById(int itemId) {
	String sql = "SELECT item_id, category, item_name, unit_price FROM items WHERE item_id = ?";

	// 接続を取得し、実行する SQL 文をバインドする（try-with-resources）
	try (Connection con = getConnection();
			PreparedStatement ps = con.prepareStatement(sql)) {

		// 先頭のプレースホルダに itemId をセットする
		ps.setInt(1, itemId);

		// 検索を実行し、結果セット（カーソル）を開く
		try (ResultSet rs = ps.executeQuery()) {
			// カーソルを1行進め、行があれば列から値を読み取り ItemDto を組み立てて返す
			if (rs.next()) {
				return new ItemDto(
						rs.getInt("item_id"),
						rs.getString("category"),
						rs.getString("item_name"),
						rs.getInt("unit_price"));
			}
			// 該当行がなければ null
			return null;
		}
	}
}

9. 最小のミッション（DAOへ切り出す）

  • 直書き JDBC を DAO クラスに移し、呼び出し側は DAO を「1行呼ぶ」状態にする
  • SQL は PreparedStatement にし、文字列連結をやめて「?」へ setXxx で値を渡す

10. 例外処理（第3回前に押さえる）

カリキュラム上は第3回の直前に、Unit90_Exception（および curriculum/units/optional-exception-handling.md）で try / catch / try-with-resources を扱う想定です。

この教材では、DAO 内で SQLException を捕捉し、まずは e.printStackTrace() で原因の見える化をします（本来はログ設計・通知・リカバリーを検討します）。第4回の rollback も、この例外の流れを押さえると読みやすくなります。
